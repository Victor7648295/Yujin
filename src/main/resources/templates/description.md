# Программная реализация и алгоритмы работы веб-системы

## Введение

Настоящий раздел посвящён описанию программной реализации разработанной
веб-системы «BSUIR TRANSFERMARKT», предназначенной для публикации,
поиска и модерации объявлений о трансферах. В разделе последовательно
рассматриваются: применённый технологический стек, общая архитектура
программного комплекса, структура программных модулей, реализованная
модель данных, подсистема информационной безопасности, а также
алгоритмы работы ключевых функциональных подсистем. Изложение строится
от общих архитектурных решений к частным алгоритмам, что позволяет
получить целостное представление о принципах построения системы и
порядке её функционирования.

## 1. Технологический стек и обоснование его выбора

Программная реализация веб-системы выполнена на платформе **Java 17**
с использованием каркаса **Spring Boot версии 2.7.18**. Выбор данной
платформы обусловлен зрелостью экосистемы Spring, наличием готовых
решений для построения многослойных веб-приложений, а также высокой
производительностью виртуальной машины Java и поддержкой
долгосрочной эксплуатации (LTS-релиз Java 17).

В качестве слоя представления применён шаблонизатор **Thymeleaf** с
расширением `thymeleaf-extras-springsecurity5`, позволяющим
непосредственно в HTML-разметке управлять видимостью элементов
интерфейса в зависимости от состояния аутентификации текущего
пользователя. Доступ к базе данных реализован посредством
**Spring Data JPA** поверх ORM-каркаса **Hibernate**; в качестве
системы управления базами данных используется **PostgreSQL**.
Подсистема безопасности построена на каркасе **Spring Security 5.7**;
для исключения шаблонного кода применена библиотека **Lombok**.
Сборка проекта осуществляется при помощи системы **Apache Maven** с
использованием вложенной обёртки `mvnw`.

Совокупность перечисленных технологий обеспечивает строгую слоистую
структуру кода, декларативное описание конфигурации, автоматическое
управление транзакциями и единообразный подход к обработке
HTTP-запросов.

## 2. Общая архитектура программного комплекса

Архитектура веб-системы соответствует классическому шаблону
**многослойного приложения** (Layered Architecture) с разделением на
слой представления, слой бизнес-логики и слой доступа к данным.
Взаимодействие слоёв осуществляется по строго заданному направлению:
HTTP-запрос принимается контроллером, делегируется сервису, который,
в свою очередь, обращается к репозиторию. Обратное направление вызовов
запрещено, что обеспечивает низкую связанность и высокую тестируемость
программных модулей.

Точкой входа в приложение служит класс `MarketPlaceApplication`,
помеченный аннотацией `@SpringBootApplication`. Все прикладные классы
размещены во вложенных пакетах `org.marketplace.marketplace.project.*`
и логически разделены следующим образом:

| Пакет | Назначение |
|-------|------------|
| `controller` | контроллеры Spring MVC, обрабатывающие HTTP-запросы |
| `service`    | классы, инкапсулирующие бизнес-логику системы |
| `repository` | интерфейсы Spring Data JPA для доступа к базе данных |
| `model`      | JPA-сущности и объекты передачи данных (DTO) |
| `config`     | конфигурационные классы и компоненты инициализации |

Шаблоны Thymeleaf размещены в каталоге `src/main/resources/templates`,
статические ресурсы (CSS, JavaScript, изображения) — в каталоге
`src/main/resources/static`. Для статического ресурса `/img/**`
переопределён обработчик ресурсов: помимо классплейсового размещения
дополнительно подключён каталог исходных файлов проекта, что
позволяет в режиме разработки немедленно отдавать загружаемые
пользователями фотографии без пересборки приложения.

## 3. Структура программных модулей

Программный комплекс реализован в виде совокупности взаимосвязанных
модулей, каждый из которых отвечает за изолированную функциональную
область. Контроллерный слой представлен следующими классами:
`TransferController` (управление объявлениями), `TransferPageController`
(публичная страница объявления), `MyTransfersController` (личный
кабинет), `ModerationController` (модерация), `CategoryController`
(управление категориями), `UserController` (администрирование
пользователей), `ProfileController` (редактирование собственного
профиля), `RegistrationController` и `LoginController` (регистрация
и вход).

Сервисный слой образован классами `TransferService`, `TransferPageService`,
`UserService`, `CategoryService` и `CustomUserDetailsService`.
Слой доступа к данным представлен интерфейсами-репозиториями
`TransferRepository`, `TransferStatusRepository`,
`TransferConditionRepository`, `UserRepository` и `CategoryRepository`,
наследующими `JpaRepository` и предоставляющими как стандартные
CRUD-операции, так и пользовательские JPQL-запросы.

Конфигурационный пакет содержит классы `SecurityConfig` (декларация
правил безопасности), `WebConfig` (правила обработки статических
ресурсов), `StringToTransferTypeConverter` (преобразователь
строкового значения формы в сущность), а также два компонента
`CommandLineRunner` — `AdminUserSeeder` и `TransferStatusSeeder`,
обеспечивающие начальное наполнение базы данных.

## 4. Модель данных

Информационная модель предметной области реализована в виде пяти
сущностей JPA, отображённых на одноимённые таблицы реляционной базы
данных. Сущность `Transfer` (таблица `transfers`) описывает
объявление о трансфере и содержит атрибуты названия, цены, региона,
категории, контактного телефона, описания, имени продавца, пути к
изображению и времени создания. Сущности `Transfer.condition` и
`Transfer.status` связаны отношениями `@ManyToOne` со справочниками
`TransferType` («Местный», «Международный») и `TransferStatus`
(`PENDING`, `APPROVED`, `REJECTED`) соответственно. Принадлежность
объявления продавцу выражена связью `@ManyToOne` с сущностью `User`.

Сущность `User` (таблица `users`) хранит учётные данные пользователя,
включая email (уникальный идентификатор для входа), хеш пароля,
имя и фамилию, роль (`USER` либо `ADMIN`) и признак активности
учётной записи. Сущность `Category` (таблица `categories`) образует
справочник категорий объявлений; следует отметить, что поле
`Transfer.category` представлено свободной строкой и не связано
внешним ключом со справочником `Category`, что позволяет создавать
объявления в категориях, ранее не зарегистрированных в системе.

Управление схемой базы данных осуществляется в режиме
`spring.jpa.hibernate.ddl-auto=update`: изменения структуры JPA-сущностей
автоматически отражаются в схеме реляционной базы данных. Начальное
наполнение справочников выполняется программно в момент запуска
приложения (см. подраздел 5).

## 5. Инициализация системы

При старте приложения Spring Boot последовательно вызывает все
зарегистрированные компоненты, реализующие интерфейс `CommandLineRunner`.
В рамках разработанной системы определены два таких компонента.
Компонент `AdminUserSeeder` проверяет наличие в таблице `users`
учётной записи администратора с адресом `admin@admin.com` и, при её
отсутствии, создаёт её с ролью `ADMIN` и хешированным паролем `admin`.
Компонент `TransferStatusSeeder` гарантирует существование в таблице
`transfer_statuses` трёх обязательных строк со статусами `PENDING`,
`APPROVED`, `REJECTED` и соответствующими человекочитаемыми описаниями.

Помимо сидеров, в конфигурационном слое регистрируется компонент
`StringToTransferTypeConverter`, реализующий интерфейс
`Converter<String, TransferType>` и обеспечивающий автоматическое
преобразование строкового значения, поступающего из HTML-формы
создания объявления, в соответствующую сущность справочника типов
трансфера.

## 6. Подсистема информационной безопасности

Подсистема безопасности реализована средствами Spring Security и
сконфигурирована классом `SecurityConfig`. В классе декларативно
описан фильтр безопасности (`SecurityFilterChain`), задающий
дифференцированные правила доступа к ресурсам системы. Анонимный
доступ разрешён к статическим ресурсам (`/css/**`, `/js/**`,
`/images/**`, `/img/**`), к страницам входа и регистрации (`/login`,
`/register`), к GET-запросам публичной витрины (`/`, `/filter`,
`/search-by-name`), к карточкам объявлений (регулярное выражение
`^/product/\d+$`) и к конечной точке AJAX-получения телефона продавца
(`/api/product/*/phone`). Доступ к административному разделу
`/admin/**` ограничен авторитетом `ROLE_ADMIN`; все остальные ресурсы
требуют аутентификации.

Аутентификация осуществляется по схеме form-login: HTML-форма
направляет POST-запрос с параметрами `email` и `password` по адресу
`/login`; в случае успеха пользователь переадресуется на главную
страницу, в случае ошибки — на `/login?error=true`. Особенностью
конфигурации является использование email-адреса в роли имени
пользователя, что достигается вызовом `usernameParameter("email")`.
Хеширование паролей выполняется реализацией
`DelegatingPasswordEncoder` (по умолчанию — BCrypt), что обеспечивает
криптографическую стойкость хранения учётных данных и
обратную совместимость с возможными альтернативными алгоритмами.

Загрузка `UserDetails` инкапсулирована в `CustomUserDetailsService`,
который по адресу электронной почты извлекает сущность `User` из
репозитория, формирует список авторитетов из единственного элемента
`ROLE_<значение поля role>` и возвращает объект `UserDetails`,
пригодный для использования стандартным провайдером аутентификации
Spring Security.

## 7. Алгоритмы работы основных подсистем

### 7.1. Алгоритм регистрации пользователя

Процедура регистрации реализована классом `RegistrationController` во
взаимодействии с методом `UserService.register`. При получении GET-запроса
по адресу `/register` контроллер формирует модель с пустым объектом
`RegistrationRequest` и возвращает шаблон формы. При получении
POST-запроса выполняется следующая последовательность действий:

1. Запускается валидация Bean Validation для полей формы.
2. Производится сравнение полей `password` и `confirmPassword`; при
   их несовпадении в результат валидации добавляется соответствующая
   ошибка, и форма перерисовывается с сохранением ранее введённых
   значений.
3. При отсутствии ошибок управление передаётся сервисному методу
   `UserService.register`, который повторно проверяет уникальность
   адреса электронной почты, шифрует пароль посредством
   `PasswordEncoder`, формирует сущность `User` с ролью `USER` и
   признаком активности `true`, после чего сохраняет её в базе
   данных.
4. В случае исключения `IllegalArgumentException` (несовпадение
   паролей либо занятость email) форма перерисовывается с
   текстом ошибки.
5. При успешной регистрации пользователь переадресуется на страницу
   `/login` с flash-сообщением об успешном завершении операции.

### 7.2. Алгоритм аутентификации

Процедура входа в систему обрабатывается стандартным фильтром
аутентификации Spring Security, активируемым POST-запросом по адресу
`/login`. Алгоритм аутентификации включает следующие шаги:

1. Метод `CustomUserDetailsService.loadUserByUsername` выполняет поиск
   пользователя в базе данных по адресу электронной почты.
2. В случае отсутствия пользователя выбрасывается исключение
   `UsernameNotFoundException`, и фильтр выполняет переадресацию на
   `/login?error=true`.
3. При нахождении пользователя сконфигурированный `PasswordEncoder`
   сравнивает введённый пароль с сохранённым хешем при помощи метода
   `matches`.
4. При совпадении формируется аутентифицированный контекст безопасности,
   и пользователь переадресуется на главную страницу; при несовпадении —
   на страницу входа с признаком ошибки.

Контроллер `LoginController` дополнительно преобразует параметры
запроса (`error`, `logout`, `registered`) в человекочитаемые сообщения,
отображаемые шаблоном формы входа.

### 7.3. Алгоритм публикации объявления о трансфере

Алгоритм создания нового объявления реализован методом
`TransferController.createProduct` совместно с методом
`TransferService.createProduct` и включает следующие этапы:

1. По идентификатору, переданному скрытым полем формы, из базы данных
   извлекается сущность продавца и устанавливается в поле
   `Transfer.user`.
2. Вызывается приватный метод `saveImage`, реализующий алгоритм
   обработки фотографии объявления (см. подраздел 7.4); при успешном
   сохранении возвращённый относительный путь записывается в поле
   `Transfer.imagePath`.
3. Если статус объявления не задан явным образом, устанавливается
   статус `PENDING` посредством вспомогательного метода
   `getOrCreateStatus`, который выполняет поиск статуса в таблице
   `transfer_statuses` и создаёт новую запись при её отсутствии.
4. Сформированная сущность сохраняется в базе данных через
   `TransferRepository.save`, после чего пользователь переадресуется
   на главную страницу.

### 7.4. Алгоритм обработки фотографии объявления

Загруженное пользователем изображение приводится к стандартному виду —
скруглённому квадрату фиксированного размера 200×200 пикселей с
радиусом скругления 40 пикселей. Алгоритм реализован в приватном
методе `TransferService.saveImage` и выполняет следующие действия:

1. Содержимое мультипарт-файла считывается посредством
   `ImageIO.read`, что обеспечивает поддержку распространённых
   растровых форматов (PNG, JPEG, GIF, BMP).
2. Создаётся целевой буферизованный образ с альфа-каналом
   (`TYPE_INT_ARGB`) указанных размеров.
3. На объекте `Graphics2D` устанавливаются режимы интерполяции
   `BILINEAR`, рендеринга `QUALITY` и антиалиасинга `ON`.
4. Композиция `AlphaComposite.Src` обеспечивает заливку
   геометрической фигурой `RoundRectangle2D` указанного размера и
   радиуса скругления, формируя маску изображения.
5. Композиция `AlphaComposite.SrcAtop` накладывает исходное
   изображение поверх маски с масштабированием до размеров цели;
   пиксели вне маски остаются прозрачными.
6. Полученное изображение сохраняется в файл `<UUID>.png` в каталоге
   `src/main/resources/static/img/`; вызывающему коду возвращается
   относительный URL вида `/img/<UUID>.png`.

При невозможности обработать загруженный файл выбрасывается
исключение `RuntimeException` с диагностическим сообщением.

### 7.5. Алгоритм модерации объявлений

Премодерация объявлений реализована в классе `ModerationController`,
доступном исключительно пользователям с ролью `ADMIN`. При запросе
страницы `/admin/moderation` сервис `TransferService` выбирает все
объявления со статусом `PENDING` посредством JPQL-производного метода
`findByStatus_StatusName`. Администратор может перейти на страницу
детального просмотра объявления и принять одно из двух решений:

1. **Одобрить** объявление — POST-запрос `/admin/moderation/approve/{id}`
   приводит к вызову метода `TransferService.approveProduct`, который
   делегирует управление приватному методу `changeStatus` с параметром
   `APPROVED`.
2. **Отклонить** объявление — POST-запрос `/admin/moderation/reject/{id}`
   вызывает `TransferService.rejectProduct`, передающий управление
   `changeStatus` с параметром `REJECTED`.

Метод `changeStatus` выполняет следующую последовательность действий:
выборку объявления по идентификатору, установку нового статуса через
`getOrCreateStatus`, сохранение обновлённой сущности и возврат
логического признака успеха. Результат операции отображается на
странице модерации в виде flash-сообщения.

### 7.6. Алгоритм фильтрации и поиска объявлений

Подсистема витрины объявлений реализует три режима отображения:
полный список одобренных объявлений, фильтрацию по совокупности
критериев и поиск по фрагменту названия. Все три режима используют
единый шаблон `index.html` и отличаются только источником данных,
помещаемых в модель.

В режиме фильтрации запрос обрабатывается методом
`TransferService.searchProducts`, который делегирует выполнение
JPQL-запросу `TransferRepository.searchProducts`. Структура запроса
построена по принципу опционального применения условий: для каждого
параметра фильтрации используется конструкция вида
`:param IS NULL OR :param = '' OR field = :param`. Подобный подход
позволяет в одном запросе обрабатывать любую комбинацию заданных и
незаданных критериев без необходимости динамического построения
запроса средствами Criteria API.

Поиск по названию реализован методом `TransferService.searchByName`,
который при наличии непустой подстроки запроса передаёт её в
JPQL-запрос `searchByTitle` с условием
`LOWER(p.title) LIKE LOWER(CONCAT('%', :query, '%'))`. Регистронезависимое
сопоставление обеспечивает корректную работу с кириллическими
символами вне зависимости от регистра ввода. Во всех режимах витрина
отображает исключительно объявления со статусом `APPROVED`, что
гарантируется фильтром `p.status.id = 8` в соответствующих
JPQL-запросах.

Для обеспечения «прилипающего» поведения формы фильтрации контроллер
помещает в модель не только результирующий набор объявлений, но и
ранее выбранные значения критериев (`selectedRegion`,
`selectedCategory`, `selectedCondition`, `priceFrom`, `priceTo`),
что позволяет шаблонизатору восстановить состояние формы при
повторной отрисовке страницы.

### 7.7. Алгоритм работы личного кабинета

Личный кабинет пользователя реализован классом `MyTransfersController`.
При запросе GET `/my-products` контроллер извлекает имя пользователя
из объекта `Principal`, представляющего текущий аутентификационный
контекст. По адресу электронной почты в репозитории отыскивается
идентификатор пользователя, после чего вызывается метод
`TransferService.getProductsByUser`, реализованный посредством
производного запроса `findByUser_IdOrderByStatus_StatusNameAsc`.
Полученный список объявлений, отсортированный по значению статуса,
помещается в модель и отображается шаблоном `my-products.html`.
В случае отсутствия аутентификационного контекста выполняется
переадресация на страницу входа.

### 7.8. Алгоритм самостоятельного редактирования профиля

Возможность редактирования собственных учётных данных предоставляется
классом `ProfileController` и сервисным методом
`UserService.updateProfile`. Алгоритм обновления включает следующие
проверки и действия:

1. По адресу электронной почты текущего пользователя извлекается
   сущность `User`.
2. Введённый текущий пароль сравнивается с сохранённым хешем
   посредством `PasswordEncoder.matches`; при несовпадении
   выбрасывается `IllegalArgumentException`.
3. При смене адреса электронной почты выполняется проверка
   уникальности нового значения.
4. Поля имени и фамилии обновляются при условии непустоты входных
   значений.
5. Если задан новый пароль, проверяется его совпадение с подтверждением,
   после чего пароль шифруется и сохраняется.

В случае изменения адреса электронной почты система выполняет
принудительный выход пользователя из системы (переадресацией на
`/logout`), поскольку имя пользователя в контексте безопасности
становится несогласованным с фактическим значением в базе данных.

### 7.9. Алгоритмы администрирования

Административная подсистема представлена тремя контроллерами:
`UserController` (управление учётными записями пользователей),
`CategoryController` (управление справочником категорий) и
`ModerationController` (рассмотрен в подразделе 7.5). Управление
пользователями реализует следующие операции: поиск по фрагменту имени
или фамилии, частичное обновление полей профиля (метод
`UserService.adminUpdate`), смену роли, сброс пароля и
блокировку/разблокировку учётной записи. Управление категориями
поддерживает полный набор CRUD-операций; для каждой операции
обеспечивается проверка уникальности названия категории, а в случае
её нарушения возбуждается исключение с соответствующим диагностическим
сообщением.

### 7.10. Алгоритм асинхронного получения телефона продавца

Для предотвращения автоматизированного сбора телефонных номеров
веб-страницами поисковых систем номер продавца не отображается в
HTML-разметке страницы объявления, а запрашивается асинхронно при
нажатии пользователем кнопки «Позвонить». Клиентский JavaScript-код
выполняет AJAX-запрос вида GET `/api/product/{id}/phone`, обрабатываемый
методом `TransferController.getPhoneNumber`. Метод извлекает объявление
из репозитория и возвращает строковое значение поля `phone` с
аннотацией `@ResponseBody`, либо пустую строку в случае отсутствия
объявления. Полученное значение отображается в модальном окне
интерфейса.

## 8. Подсистема обработки статических и пользовательских ресурсов

Особенностью реализации является переопределение стандартного
обработчика статических ресурсов в классе `WebConfig`. Для URL-шаблона
`/img/**` декларируются две локации ресурсов в следующем порядке
приоритета: каталог исходных файлов проекта
`file:src/main/resources/static/img/` и каталог классплейса
`classpath:/static/img/`. Подобная конфигурация позволяет немедленно
отдавать загружаемые в режиме разработки фотографии (которые
физически записываются в исходный каталог методом `saveImage`)
без необходимости пересборки приложения. Сидовые изображения,
поставляемые вместе с приложением, остаются доступны из классплейса.

## 9. Сквозные соглашения о потоках управления

Программная реализация системы придерживается следующих сквозных
соглашений, обеспечивающих единообразие поведения контроллеров и
предсказуемость пользовательского опыта.

Во-первых, все контроллеры возвращают либо имя Thymeleaf-шаблона
(в случае рендеринга страницы), либо префиксированную строку
`redirect:` (в случае пост-редирект-гет-сценария). Применение
PRG-схемы предотвращает повторную отправку POST-запроса при
обновлении страницы браузером.

Во-вторых, сообщения об успехе и ошибке передаются через flash-атрибуты
объекта `RedirectAttributes`, что обеспечивает их однократное
отображение после переадресации и автоматическое удаление из
сессии.

В-третьих, бизнес-исключения представлены классом
`IllegalArgumentException` с понятным локализованным сообщением;
исключения перехватываются контроллером и преобразуются в
flash-атрибут `errorMessage`, отображаемый шаблоном в виде
информационного блока.

В-четвёртых, формы фильтрации реализуют sticky-поведение: контроллер
помещает в модель выбранные пользователем значения критериев, что
позволяет шаблонизатору восстановить состояние элементов управления
при повторной отрисовке страницы.

## 10. Выводы

В рамках настоящего раздела дано систематическое описание программной
реализации разработанной веб-системы. Применённые архитектурные решения
(многослойная структура, разделение ответственности между слоями,
декларативная конфигурация безопасности) и технологические средства
(Spring Boot, Spring Security, Spring Data JPA, Thymeleaf, PostgreSQL)
обеспечивают сопровождаемость кода, безопасность хранения учётных
данных и гибкость в части последующего развития функциональности.
Реализованные алгоритмы, охватывающие весь жизненный цикл объявления
о трансфере — от регистрации пользователя и публикации объявления до
его модерации и предъявления в публичной витрине, — отвечают
требованиям технического задания и образуют целостную совокупность
взаимосвязанных функциональных подсистем.
